# Customer skins

This directory holds the customer-skin configs that re-theme the demo from
"SLB-pattern" to "Halliburton-pattern" (or any other tier-one OFS major) by
flipping one env var. The principle from `SPECS.md` is **configuration over
code**: brand, persona names, hero asset taxonomy, customer-account names,
KPI labels, and terminology all live here — not in agent or canvas code.

## Anatomy of a skin

```
skins/<slug>/
├── customer.yaml          # the contract — validated against schema/customer.schema.json
└── assets/
    ├── logo.svg           # small brand mark (used in the canvas chrome)
    ├── hero.svg           # placeholder hero rect (kept as SVG even with .svg extension; rasterize at deploy if needed)
    └── colors.json        # primary/secondary/accent for design tooling — derived from customer.yaml
```

`customer.yaml` schema lives at `skins/schema/customer.schema.json` and is
validated by `agents/tests/unit/test_skin_schema.py`.

## Selecting the active skin

Skin selection happens at build time. The canvas reads
`NEXT_PUBLIC_CUSTOMER_SKIN` from the environment; `scripts/compile_skin.py`
compiles the YAML into `canvas/src/data/skin.generated.ts` (a typed TS
constant) before `next build` runs.

```bash
# Compile the skin into the canvas data dir
venv-deploy-310/bin/python scripts/compile_skin.py --skin halliburton

# Then build
cd canvas && NEXT_PUBLIC_CUSTOMER_SKIN=halliburton npm run build
```

The Makefile wraps both steps in `make use-skin`:

```bash
make use-skin SKIN=halliburton
make use-skin SKIN=default
```

The default skin is what fresh clones of the repo build with — no env var
required.

## Frontend integration

* `canvas/src/data/skin.generated.ts` — typed TS const, regenerated by
  `compile_skin.py`. Do not edit by hand. Committed so fresh clones build
  cleanly without needing to run the compiler first.
* `canvas/src/lib/skin.ts` — re-exports the const plus typed helpers
  (`getActiveSkin`, `getPersona`, `getScenario`).
* `canvas/src/hooks/useSkin.ts` — React hook returning the loaded skin.
* `canvas/src/data/personas.ts` — wrapper that derives the persona list
  from the active skin (so `/demo` and the cargo-plane page get re-skinned
  for free).

## Authoring a new skin

1. Copy an existing skin folder:
   ```bash
   cp -r skins/default skins/baker-hughes
   ```
2. Edit `skins/baker-hughes/customer.yaml`:
   * `meta`: slug, colors, names
   * `personas`: 6 entries; `id` values are fixed (`david`, `tomas`, `maria`, `priya`, `rafael`, `ayesha`)
   * `taxonomy.hero_asset`: customer-specific canonical id + label
   * `scenarios.cargo-plane`: location, customer account, costs
   * `terminology`: OCC vs MOC, fleet unit terminology, etc.
   * `kpi_labels`: customer-specific KPI naming
3. Drop new `logo.svg` and `hero.svg` into `assets/`.
4. Validate:
   ```bash
   venv-deploy-310/bin/python -m pytest agents/tests/unit/test_skin_schema.py -v
   venv-deploy-310/bin/python scripts/compile_skin.py --skin baker-hughes
   make use-skin SKIN=baker-hughes
   cd canvas && npm run build
   ```
5. Commit `skins/baker-hughes/` and (only if running locally with the new
   skin active) the regenerated `canvas/src/data/skin.generated.ts`.

## What can vary across skins

Everything in `customer.yaml`: brand, 6 persona names + roles + regions,
hero/secondary asset taxonomy, scenario-specific customer accounts and
locations, terminology glossary, KPI labels.

## What cannot vary

These are structural and require code changes, not skin changes:

* The 6 persona IDs and their S&OP stages.
* The Workflow graph (Capacity Orchestrator's Workflow nodes).
* The Knowledge Catalog Aspect Type schema.
* The MCP server tool surface.
* The governance posture (Identity, Gateway, Model Armor configuration).
